# Week 5d-1: Epoch Order Accumulator Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make `submit_order` and `cancel_order` write per-order commitments to two on-chain running-hash chains (`order_acc`, `cancel_acc`) per epoch, so the future Week 5d-2 clearing circuit can bind its private order list to on-chain state.

**Architecture:** Extend the `EpochState` `PublicMutable` struct with four fields (`order_acc`, `cancel_acc`, `order_count`, `cancel_count`). Two new `#[only_self]` public callbacks (`_append_order`, `_append_cancel`) fold a Poseidon2-hashed order commitment into the appropriate chain. `submit_order` and `cancel_order` compute the commitment in private context and enqueue the callback; existing W5c enqueue ordering (guards before escrow-touching calls) is preserved.

**Tech Stack:** Noir (aztec-nr v4.2.0), Aztec.js v4.2.1, `@aztec/foundation/crypto/poseidon`, TypeScript `node:test` + `tsx` integration runner, TXE for in-process Noir tests.

**Source spec:** `docs/superpowers/specs/2026-05-20-zswap-aztec-week-05d-1-order-accumulator-design.md`

```noir
    /// Public callback used by `submit_order` to fold an order commitment into the
    /// epoch's `order_acc` running-hash chain and to enforce the per-epoch cap.
    /// `only_self` keeps this callable only via this contract's `enqueue_self`.
    ///
    /// Capacity revert message is the user-visible signal that 128 orders have been
    /// submitted in the current epoch. The whole tx (including the escrow and the
    /// `OrderNote` insertion) reverts because all callbacks share atomicity with
    /// the private body that enqueued them.
    #[external("public")]
    #[only_self]
    fn _append_order(c_i: Field) {
        let mut epoch = self.storage.current_epoch.read();
        assert(
            epoch.order_count < MAX_ORDERS_PER_EPOCH,
            "epoch order capacity reached",
        );
        epoch.order_acc = Poseidon2::hash([epoch.order_acc, c_i], 2);
        epoch.order_count = epoch.order_count + 1;
        self.storage.current_epoch.write(epoch);
    }
```

- [ ] **Step 3: Modify `submit_order` to compute `c_i` and enqueue `_append_order`.**

In `contracts/orderbook/src/main.nr`, replace the tail of `submit_order` — everything from the `OrderNote` insertion through the final closing brace (currently lines 183-201). The replacement keeps the existing `OrderNote` insertion and the `_assert_epoch_open` enqueue, and adds the commitment computation + `_append_order` enqueue:

```noir
        // Build the OrderNote and insert it into the maker's private set, with on-chain
        // constrained encrypted delivery so the maker is guaranteed to discover and decrypt it.
        let submitted_at_block: u32 = self.context.get_anchor_block_header().block_number();
        let order = OrderNote {
            submitted_at_block,
            side,
            amount_in,
            limit_price,
            nonce: order_nonce,
            owner: maker,
        };
        self.storage.orders.at(maker).insert(order).deliver(MessageDelivery.ONCHAIN_CONSTRAINED);

        // Per-order commitment bound to the full OrderNote field set (the Week 5d-2 clearing
        // circuit will reproduce this from its private order list). `order_nonce` is a random
        // Field, so c_i is hiding: side/amount/limit cannot be recovered from c_i.
        let c_i: Field = Poseidon2::hash(
            [
                maker.to_field(),
                if side { 1 } else { 0 },
                amount_in as Field,
                limit_price as Field,
                order_nonce,
                submitted_at_block as Field,
            ],
            6,
        );

        // Defer to public execution: epoch-open check first (clean revert message if the
        // epoch has advanced), then the accumulator append (also revertible on capacity).
        // Both callbacks run atomically with this private body; any failure reverts escrow
        // and the OrderNote together.
        self.enqueue_self._assert_epoch_open();
        self.enqueue_self._append_order(c_i);
    }
```

(Note: `AztecAddress::to_field()` is the aztec-nr method that exposes the address's inner `Field`. The `if side { 1 } else { 0 }` form sidesteps any ambiguity in `bool → Field` casting and is explicit at the call site.)

    /// Public callback used by `cancel_order` to fold a cancelled-order commitment
    /// into the epoch's `cancel_acc` running-hash chain. `only_self`.
    ///
    /// No cap check: `cancel_count` only advances when a real `OrderNote` was popped
    /// in `cancel_order`, and `OrderNote` insertion is capped to MAX_ORDERS_PER_EPOCH
    /// by `_append_order`, so `cancel_count ≤ order_count ≤ MAX_ORDERS_PER_EPOCH`
    /// holds inherently.
    #[external("public")]

    fn _append_cancel(c_i: Field) {
        let mut epoch = self.storage.current_epoch.read();
        epoch.cancel_acc = Poseidon2::hash([epoch.cancel_acc, c_i], 2);
        epoch.cancel_count = epoch.cancel_count + 1;

```ts
/**
 * Mirror of `Orderbook::_append_order`'s c_i computation. Inputs MUST be in the
 * exact field order the contract uses; otherwise the recomputed acc' will not match.
 */
async function orderCommitment(args: {
  owner: AztecAddress;
  side: boolean;
  amountIn: bigint;
  limitPrice: bigint;
  orderNonce: bigint | Fr;
  submittedAtBlock: number;
}): Promise<Fr> {
  return poseidon2Hash([
    args.owner.toField(),
    args.side ? 1n : 0n,
    args.amountIn,
    args.limitPrice,
    typeof args.orderNonce === "bigint" ? args.orderNonce : args.orderNonce.toBigInt(),
    BigInt(args.submittedAtBlock),
  ]);
}

/** acc' = poseidon2([acc, c_i]); the running-hash fold used by both append callbacks. */
async function foldChain(acc: Fr, c: Fr): Promise<Fr> {
  return poseidon2Hash([acc, c]);
}
```
